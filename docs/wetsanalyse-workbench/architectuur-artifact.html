Een werkbank waarin de jurist wetsartikelen ophaalt en volgens het Juridisch Analyseschema annoteert — handmatig én met een AI-agent die elk voorstel ter menselijke beslissing voorlegt, alles brongetrouw en met een onwrikbare audit trail.

Wat we bouwen

Een chat-scherm in ChatGPT/Claude-stijl waarin je de multi-agent aanspreekt, met links een documentpaneel dat het opgehaalde artikel toont. Op die tekst annoteer je volgens het Juridisch Analyseschema (JAS). De workbench kent twee samenwerkende modi.

Modus 1 · de jurist stuurt

Handmatig annoteren

Selecteer een tekstfragment, ken een JAS-klasse + toelichting toe; onder de tekst groeit een annotatietabel (klasse · fragment · toelichting · vindplaats). Vraag de agent desgewenst als sparringpartner om suggesties op je selectie.

Modus 2 · de agent stelt voor

Agent-annotatie met HITL

De agent voert zelf een JAS-annotatie uit en biedt elk element afzonderlijk aan voor een menselijke beslissing — approve · edit · reject · comment — met per beslissing een lesson learned. Elke stap belandt in de audit trail.

Kern voor kwaliteit

De leerlus — beter zonder hertrainen

Menselijke correcties zijn geen wegwerpdata. Ze worden opgeslagen en als few-shot-context per JAS-klasse en werkgebied teruggevoed naar de agent. Approved annotaties verrijken de graaf, die de agent opnieuw voedt. Data groeit, de agent wordt beter.

Het onderscheidende

Review-workflow — de ervaring is het product

Het verschil zit niet in "nog slimmere AI", maar in een review-ervaring die aanvoelt als GitHub pull requests, VS Code, Figma en Copilot samen: de AI doet een voorstel, de mens beslist — sneller en consistenter. HITL is daarom geen enkele knop maar een meertraps workflow.

Critic-agent

Een aparte agent controleert vóór de jurist: ontbrekende elementen, verkeerde klasse, inconsistentie, zwakke bron. Scheelt reviewwerk; past in het multi-agent-patroon.

Disambiguatie

Bij twijfel geen cijfer, maar alternatieven met motivatie — de jurist kiest. Maakt de interpretatiekeuze expliciet, kernwaarde van de methode.

Missing-elements

Meldt óók wat waarschijnlijk ontbreekt (uitzondering, definitie, verwijzing, overgangsrecht) o.b.v. vergelijkbare artikelen — als aan te vinken suggestie, nooit als bewering.

Element-diff

Na een edit een veld- en tekst-diff (+ toegevoegd / − verwijderd / ~ gewijzigd) — de correctie-delta die de jurist ziet én die de lessons voedt.

Review-queue + tijdlijn

Rechts een reviewpaneel met statustelling (akkoord / controleren / ontbreekt); de audit als leesbare tijdlijn (actor · actie · tijd).

Confidence — met zorg

Geen rauwe modelfloats (slecht gekalibreerd → schijnprecisie). In plaats daarvan een aandacht-niveau 🟢🟡🔴 afgeleid van echte signalen (critic-flag, alternatieven, grounding, patroon-afwijking).

Datamodel · api-contracten

Wat een annotatie is

AnnotatieElement

id · klasse (1 van 13) · tekst · span [start,end] · vindplaats · toelichting · herkomst (mens|agent) · lifecycle (concept→ai_proposed→critic_checked→human_approved→published→reused) · alternatieven[] · aandacht (🟢🟡🔴) · beslissingen[] (+ review_reason) · diff · lessons[].

AnnotatieDocument

Groepeert elementen per bron (bwbId + artikel[+lid]) binnen een werkgebied; status in-bewerking → geaccordeerd → gepromoveerd. Draagt de audit trail.

AuditRecord · append-only

Elke mutatie en beslissing onwrikbaar vastgelegd: wie, wat, wanneer, waarom. Nooit overschreven — de juridische verantwoording.

Lessons-store

Geaggregeerde leerpunten per JAS-klasse/werkgebied, opvraagbaar door de agent als few-shot-voorbeelden bij nieuwe voorstellen.

Uitvoering

Fasering

We beginnen met dit document, dan het fundament, dan een direct bruikbare handmatige workbench — en bouwen de agent, de diepere JAS-laag en de graaf-promotie daar bovenop.

Deliverable 0 Architectuur-artifact dit document

De volledige visie, architectuur, datamodel en fasering — visueel en deelbaar — ter beoordeling vóór er code komt.

Fase 0 Fundament & datamodel api

JAS-klassen-referentie uit de bron; tabellen + contracten (element/document/audit/lessons) + skelet-endpoints; auth hergebruikt. Append-only audit vanaf dag één.

Fase 1 Workbench-UI + handmatige annotatie MVP

Nieuw scherm: chatpaneel (bestaande QA-agent) + documentpaneel + selectie → klasse + toelichting → annotatietabel; persistent met audit. Rijkshuisstijl + JAS-kleuren.

Fase 2 Annotatie-agent + Critic + review-workflow graph-qa

annotatie-specialist uit de JAS-bron + een Critic-agent; review-queue met decision-cards, disambiguatie, missing-elements en diff; lessons-learned die de agent terugvoedt.

Fase 3 Diepere JAS · activiteit 3

Van markeren naar begrippen + afleidingsregels, methode-getrouw; werkgebied-brede ontdubbeling. Zelfde HITL + audit.

Fase 4 Promoveren naar de graaf api

JAS-annotatie-vocabulaire; één geauthenticeerd, idempotent, geaudit schrijfpad. Adresseert meteen het open+writable-graaf-risico. Daarna bevraagbaar door de QA-agent.

Niet onderhandelbaar

Principes

  • Brongetrouwheid. Elk element herleidbaar naar bwbId + artikel/lid + tekst-span.
  • Mens beslist. Een agent-voorstel is nooit definitief zonder approve/edit.
  • Vers vanaf de bron. De JAS-methode uit docs/wetsanalyse/, niet de bestaande skill.
  • Conform de agent. Specialist in de multi-agent · SSE · getypeerde tools · observability.
  • Auditbaar. Elke beslissing onwrikbaar, van begin af aan.
  • Graaf-veilig. Read-only agent; één gecontroleerd schrijfpad.

Per fase

Verificatie

  • Unit-tests met DI/fakes.
  • Een offline, gescript end-to-end pad.
  • Live rooktest via de workbench.
  • Geen graaf-mutatie vóór Fase 4.
  • Audit-integriteit + span↔bron expliciet getest.
Elke fase apart: branch → PR → merge → deploy, conform de huidige werkwijze.
